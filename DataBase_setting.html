数据库配置与前端发布指南

使用 Supabase 作为后台数据库 + GitHub Pages 发布前端网页

方案概述 Supabase 数据库 前后端关联 踩坑记录 GitHub Pages 发布 验证与迭代

方案概述

本指南介绍如何为 HTML 前端页面接入 Supabase 后台数据库，并通过 GitHub Pages 将网页发布上线。整体架构如下：

前端层 — 纯 HTML/CSS/JS 静态页面，无后端服务

数据库层 — Supabase（基于 PostgreSQL，提供 REST API + SDK）

托管层 — GitHub Pages（免费 HTTPS + 自定义域名支持）

部署流程 — AI 生成 HTML => 提交到 GitHub => GitHub Pages 自动部署

一、Supabase 数据库配置

1.1 注册 Supabase

访问 supabase.com，使用 GitHub 账号登录或邮箱注册。

Supabase 提供免费套餐（Free Tier）：

  • 500MB 数据库空间
  • 50,000 月活跃用户
  • 2GB 带宽
  • 支持 PostgreSQL 所有核心功能

1.2 创建项目

1 新建项目

登录后点击「New Project」，填写：

  • Name：项目名称（如 voting-system）
  • Database Password：设置数据库密码（务必记住）
  • Region：选择距离最近的区域（如东京 ap-northeast-1）
  • Pricing Plan：选择 Free Tier 即可

2 等待初始化

创建后需要几秒钟自动初始化。完成后进入项目仪表盘。

1.3 创建数据表

在项目仪表盘中，可以创建数据库表。以投票系统为例：

方法一：使用 SQL 编辑器

进入 SQL Editor，执行如下建表语句：

-- 活动表
CREATE TABLE activities (
  id BIGINT GENERATED ALWAYS AS IDENTITY PRIMARY KEY,
  title TEXT NOT NULL,
  description TEXT,
  start_time TIMESTAMPTZ,
  end_time TIMESTAMPTZ,
  status TEXT DEFAULT 'draft',
  created_at TIMESTAMPTZ DEFAULT NOW()
);

-- 报名表
CREATE TABLE participants (
  id BIGINT GENERATED ALWAYS AS IDENTITY PRIMARY KEY,
  activity_id BIGINT REFERENCES activities(id),
  name TEXT NOT NULL,
  phone TEXT,
  email TEXT,
  created_at TIMESTAMPTZ DEFAULT NOW()
);

-- 投票表
CREATE TABLE votes (
  id BIGINT GENERATED ALWAYS AS IDENTITY PRIMARY KEY,
  activity_id BIGINT REFERENCES activities(id),
  participant_id BIGINT REFERENCES participants(id),
  voter_id TEXT NOT NULL,
  created_at TIMESTAMPTZ DEFAULT NOW(),
  UNIQUE(activity_id, voter_id)
);

方法二：使用 Table Editor 可视化创建

进入 Table Editor（左侧菜单），点击「Create a new table」，逐字段添加：

  • 字段名称、类型、默认值
  • 主键、外键约束
  • 唯一约束（如投票防重复）

1.4 获取 API 密钥

在项目仪表盘左侧 → Project Settings → API：

  • Project URL：你的 Supabase 项目地址
  • anon/public API key：前端使用的公开 API Key
  • service_role key：管理员专用，不要暴露在前端

复制 Project URL 和 anon key，接下来在 HTML 中配置。

二、HTML 前端连接 Supabase

2.1 引入 Supabase SDK

在 HTML 的 <head> 中引入 Supabase JavaScript 客户端：

<script src="https://unpkg.com/@supabase/supabase-js@2"></script>

或者使用 ES Module 方式：

<script type="module">
  import { createClient } from 'https://esm.sh/@supabase/supabase-js@2'
</script>

2.2 初始化客户端

// 替换为你的项目信息
const SUPABASE_URL = 'https://xxxxxxxxxxxx.supabase.co'
const SUPABASE_ANON_KEY = 'eyJxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxx'

const supabase = createClient(SUPABASE_URL, SUPABASE_ANON_KEY)

2.3 常用数据库操作

查询数据

const { data, error } = await supabase
  .from('activities')
  .select('*')
  .eq('status', 'active')
  .order('created_at', { ascending: false })

插入数据（报名）

const { data, error } = await supabase
  .from('participants')
  .insert([
    { activity_id: 1, name: '张三', phone: '138xxxx' }
  ])
  .select()

投票（含重复检查）

const { data, error } = await supabase
  .from('votes')
  .insert([
    { activity_id: 1, participant_id: 3, voter_id: 'user_001' }
  ])
  .select()

// 如果返回 23505 错误（唯一约束冲突），说明已投票

实时排名

const { data, error } = await supabase
  .from('votes')
  .select('participant_id, count')
  .eq('activity_id', 1)
  .order('count', { ascending: false })

安全提示

  • API Key 暴露在前端是安全的，因为 Supabase 通过 Row Level Security（RLS）控制权限
  • 默认情况下，建议启用 RLS 并设置合理的访问策略
  • 在 SQL Editor 中运行：ALTER TABLE your_table ENABLE ROW LEVEL SECURITY;

2.4 允许跨域访问

进入 Supabase 项目设置 → API → 将你的 GitHub Pages 域名添加到允许列表：

https://your-username.github.io

三、踩坑记录：Supabase + Vercel

我在最初阶段尝试了 Supabase + Vercel 的方案，发现了两个关键问题。

问题一：Vercel 在中国大陆无法访问

Vercel 是一个优秀的全球前端部署平台，支持 GitHub 自动部署、HTTPS、自定义域名。但它的 CDN 节点在中国大陆被屏蔽，用户直接访问会超时或无法连接。

解决方案：使用 GitHub Pages 替代 Vercel。GitHub Pages 在国内大多数网络环境下可以访问。

问题二：Supabase API 国内访问不稳定

Supabase 的 API 服务器在美国/欧洲，从中国大陆直接请求 API 有时会出现高延迟或偶尔超时。如果你的项目面向国内用户，这是需要评估的点。

缓解方案：

  • 创建项目时选择东京（Tokyo）区域，延迟较低
  • 前端合理使用数据缓存，减少重复请求
  • 如果必须面向国内生产环境，考虑国内云数据库方案

经验总结

技术方案没有完美的。Supabase + Vercel 对于海外用户或开发者演示非常好用（免费、无需维护、功能强大），但对于需要稳定服务中国用户的场景，需要做好应对方案。

四、GitHub Pages 发布前端

4.1 创建 GitHub 仓库

1 登录 GitHub

访问 github.com 并登录。

2 创建新仓库

点击右上角 + 号 → New repository：

  • Repository name：如 voting-system
  • Public / Private：选 Public（GitHub Pages 免费版需要 Public 仓库）
  • 勾选「Add a README file」（可选）

4.2 提交 HTML 文件

方式一：通过 Git 命令行

git init
git add .
git commit -m "初始化投票系统"
git branch -M main
git remote add origin https://github.com/用户名/voting-system.git
git push -u origin main

方式二：直接在 GitHub 网页上传

进入仓库 → Add file → Upload files → 将 HTML 文件拖拽上传 → Commit changes。

方式三：通过 OpenClaw 自动提交（专家模式）

在 OpenClaw 中配置 GitHub 仓库后，只需对 AI 说一句话，AI 会自动：

  • 修改 HTML 文件内容
  • 执行 git add / git commit / git push
  • 触发 GitHub Pages 自动部署

整个流程：一句话搞定！

4.3 启用 GitHub Pages

1 进入仓库设置

在仓库页面点击 Settings → 左侧 Pages。

2 配置部署源

在「Build and deployment」中：

  • Source：选择「Deploy from a branch」
  • Branch：选择 main，目录选 / (root)
  • 点击 Save

3 等待部署

保存后 GitHub 会自动开始构建和部署。几分钟后会显示：

Your site is live at https://你的用户名.github.io/voting-system/

HTML 文件名要求

  • 首页文件必须命名为 index.html，否则访问根路径不会自动显示
  • 其他页面文件可以直接通过文件名访问，例如 /vote.html
  • 支持子目录结构，如 /pages/about.html

4.4 绑定自定义域名（可选）

如果你有自己的域名，可以绑定到 GitHub Pages：

  • 在仓库 Settings → Pages → Custom domain 中输入你的域名
  • 在域名 DNS 管理中添加 CNAME 记录指向 你的用户名.github.io
  • 勾选 Enforce HTTPS（会自动签发 SSL 证书）

五、验证与持续迭代

5.1 端到端验证

部署完成后，按以下流程验证：

  • 浏览器打开 GitHub Pages 地址 → 确认页面正常显示
  • 提交一条投票数据 → 检查数据是否成功写入 Supabase
  • 刷新页面 → 确认排名和投票结果正确读取
  • 使用不同浏览器/设备测试 → 确认跨平台兼容性

5.2 快速迭代流程

借助 OpenClaw，迭代可以做到：

  • 在飞书中对 AI 说：「帮我改一下投票按钮的颜色」
  • AI 修改 HTML → git push → GitHub Pages 自动部署
  • 整个过程 不到 1 分钟，无需手动操作

对比传统模式：改代码 → 本地 build → 手动上传 → 等生效。至少 10 分钟。

最终推荐

对于需要在中国大陆稳定运行的投票系统，推荐 Supabase（东京区域）+ GitHub Pages 的组合。这是我自己经过多次踩坑后验证的最佳方案。

如果与 OpenClaw 配合，还可以实现「修改 HTML → 自动 git push → 自动部署」的闭环，真正做到一句话完成发布。
